The runtime agent is how the panel sees inside a running server. It has three parts:

1. **The supervisor** - `cmd/runtime`, PID 1 in every server container. Runs java as a child process and watches it.
2. **The disco-agent mod** - a small Java mod/plugin (`agent/`) the supervisor installs on supported loaders. Reports what only in-game code can see: tick timing, player events, world stats.
3. **The panel hub** - `internal/agent`, the panel-side endpoint. Feeds everything into the existing metrics collector and event bus. No parallel state, no second bus.

There is one channel to the panel, owned by the supervisor. The mod never talks to the panel directly - it talks to the supervisor over loopback, and the supervisor relays. This keeps gRPC out of the JVM entirely, which matters because Minecraft bundles its own Netty and grpc-java's transports are a known conflict tarpit.

## The supervisor

The entrypoint stays resident instead of exec'ing java (see [Runtime Image](/development/runtime-image/) for what else it does at boot). Supervising the child buys, with zero mods installed:

- **Console over stdin** - commands work during boot and with RCON disabled. The panel's command sender prefers RCON (it captures the response) and falls back to agent stdin whenever RCON can't serve.
- **Ready detection** - the vanilla/Paper/Forge/NeoForge "Done (9.418s)!" console line, or the mod's server-started hook, whichever fires first. Replaces waiting out the SLP grace window on slow modpacks.
- **Crash forensics** - exit code, the newest crash report under `crash-reports/`, and a capped excerpt, all reported before the container exits.
- **Signal forwarding and exit codes** - SIGTERM reaches java for the graceful world save, and the child's exit code is propagated so docker restart policies keep working.
- **Process telemetry** every 15 seconds: CPU and RSS from `/proc`, CFS throttling counters and the CPU quota from the cgroup (v1 and v2), and GC pauses parsed from the JVM's unified GC log, which is always enabled on Java 11+ (rotated, size-capped).

## The session to the panel

`proto/discopanel/agent/v1` defines one service with one RPC: a bidirectional `Session` stream. Telemetry flows up, console commands and chat messages flow down. The panel already serves Connect over h2c on its one port, so the stream needs no extra listener.

The container dials the panel, not the other way around. That survives container IP churn and panel restarts (the supervisor redials with capped backoff, and h2 pings detect half-dead connections). On reconnect it resends the ready state and the command list. When the panel is unreachable, telemetry is dropped rather than queued - the server always comes first.

Authentication is a per-server bearer token:

- Before each start, `lifecycle.writeAgentSpec` generates a fresh token, stores only its SHA-256 hash on the Server row, and writes the plaintext to `/data/.discopanel/agent.json` (mode 0600) along with the panel URL.
- The panel URL is `docker.agent_url` if set, otherwise auto-detected: the panel's own container IP on the shared bridge network when the panel runs containerized, or the network gateway when it runs on the host.
- The stream handler authenticates in-handler, because streaming RPCs bypass the unary auth interceptors and agent tokens are container credentials, not user identities. The first message must be a `Hello` whose server id matches the token.

A failure to write the spec is non-fatal. The server starts anyway and monitoring falls back to scraping.

## The disco-agent mod

`agent/` is a Gradle multi-project: a shared `core` plus thin per-loader shims, the spark/FTB Essentials shape. The core compiles to Java 8 bytecode and contains the protocol client, tick recorder, and JVM sampler. Shims wire loader events into it:

| Shim | Covers | Installed into |
|---|---|---|
| `fabric` | Fabric and Quilt | `mods/` |
| `neoforge` | NeoForge | `mods/` |
| `forge` | legacy Forge (1.20.1 era) | `mods/` |
| `paper` | Paper, Purpur, Pufferfish, Spigot, Bukkit | `plugins/` |

The mod-to-supervisor link is loopback TCP on port 25585 (advertised to the JVM as `-Ddiscopanel.agent.port`), carrying 4-byte length-prefixed protobuf frames with the same `AgentMessage` envelope as the panel session. Messages the supervisor doesn't consume itself are relayed upstream unchanged. The protobuf runtime is javalite, generated by `make gen` via `buf.gen.agent.yaml` into `agent/core/src/generated`, and relocated under `app.discopanel.agent.libs` in the shipped jars so it can never clash with a pack's own protobuf.

The supervisor installs the loader-matched jar as `mods/disco-agent.jar` or `plugins/disco-agent.jar` at boot and removes it when the agent is disabled. Excluded by design: Java below 17, Folia (its regionized scheduler is incompatible with the Bukkit scheduler the paper shim uses), vanilla, and unknown loaders - all of which still get full supervisor telemetry. One known gap: Fabric has no advancement hook, so advancement events come from every platform except Fabric.

`make agent` builds the jars in a containerized Gradle (run `make gen` first, the core compiles the generated code). `docker/Dockerfile.runtime` builds them in its own stage and bakes them into `/opt/discopanel/agent/`, generating the protos itself so image builds need no local `make gen` - that's what keeps the `mod-*` tag builds in CI working from a bare checkout. Shipping a mod change means shipping runtime images.

## What flows

Up (agent to panel): `Ready`, `Stopping`, `Exited` (with crash report), `ProcSample` (CPU, RSS, throttling, GC log pauses), `TickSample` (TPS, MSPT avg/max/p95, per-dimension), `JvmSample` (heap, GC pauses from MXBeans, threads), `PlayerEvent` (join, leave, death, advancement, chat), `WorldStats` (entities, chunks, authoritative roster), `CommandList` (root command names for console autocomplete).

Down (panel to agent): `ConsoleCommand` (written to java stdin by the supervisor) and `ChatMessage` (broadcast in game by the mod).

## Precedence over scraping

Agent data lands in the same `ServerMetrics` store the SLP loop feeds, and while fresh it wins:

- Tick telemetry is the only TPS/MSPT source. RCON is never polled for metrics, it stays reserved for admin commands.
- A roster fresher than 90 seconds keeps SLP samples from clobbering player data (SLP truncates the sample) and turns off roster diffing - the hub emits exact `PLAYER_JOIN`/`PLAYER_LEAVE` events instead, plus the death/advancement/chat events scraping never had.
- The ready report counts as proof of life in the health record, so servers whose SLP is blocked go healthy immediately, and a live ready session keeps vouching while SLP pings fail.
- SLP keeps its jobs as the health probe and the source of MOTD, version, and favicon. Docker stats keep the whole-container CPU/memory view.

The degradation ladder:

1. Mod present: full game telemetry plus JVM internals.
2. Supervisor only (vanilla, or the mod failed to load): GC log, proc/cgroup stats, ready detection, stdin console. No TPS.
3. Neither (agent disabled): SLP sampling and docker stats only.

## The autopilot

`internal/autopilot` turns the telemetry plus the server's configuration into the **health check**: plain-language findings, each with a one-click fix where one exists. It is pure analysis - reads the collector's cached state, and fixes just mutate a `ServerConfig` for the caller to persist. Served by `GetServerPerformanceReport` and `ApplyPerformanceFix` on the server service, rendered as the health strip of the Server Status card on the server dashboard.

The checks: heap vs container limit (warning when headroom is thin, critical when the heap can't fit), Aikar/MeowIce both enabled, ZGC suggestion for 12 GB+ heaps on Java 21+, CPU throttling (warning at 5% of CFS periods, critical at 20%), GC pauses (warning at 200 ms, critical at 1 s), tick health from MSPT (warning at 40 ms, critical at 50), heap over 92% full, and a crash within the last 24 hours.

Two enforcement points back the report up at start time:

- The **memory guardrail** (`lifecycle/guardrail.go`) clamps a heap that can't fit the container limit for that start only, logging to the server console. The stored config is untouched - the health check points at the real fix.
- `-XX:ActiveProcessorCount` is pinned to the CFS quota by the supervisor, so GC and compiler thread pools stop being sized for the host's cores.
