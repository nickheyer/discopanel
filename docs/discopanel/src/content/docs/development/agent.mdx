The runtime agent is how the panel sees inside a running server. It has three parts:

1. **The supervisor** - `cmd/runtime` in the discoruntime repo, PID 1 in every server container. Runs java as a child process, watches it, and parses its console into events.
2. **The disco-agent javaagent** - one small jar (`agent/`, also discoruntime) attached to the server JVM via `-javaagent`. Pure JVM APIs, no Minecraft classes, no loader classes, so the same jar works on every loader, every version, and vanilla. Reports what only in-process code can see: tick thread cadence and JVM internals.
3. **The panel hub** - `internal/agent`, the panel-side endpoint. Feeds everything into the existing metrics collector and event bus. No parallel state, no second bus.

There is one channel to the panel, owned by the supervisor. The javaagent never talks to the panel directly - it talks to the supervisor over loopback, and the supervisor relays. This keeps gRPC out of the JVM entirely, which matters because Minecraft bundles its own Netty and grpc-java's transports are a known conflict tarpit.

## The supervisor

The entrypoint stays resident instead of exec'ing java (see [Runtime Image](/development/runtime-image/) for what else it does at boot). Supervising the child buys, with nothing in the JVM:

- **Console over stdin** - commands work during boot and with RCON disabled. The panel's command sender prefers RCON (it captures the response) and falls back to agent stdin whenever RCON can't serve.
- **Ready detection** - the "Done (9.418s)!" console line every loader prints.
- **Player events** - the supervisor parses the vanilla console lines every loader and version emits: joins and leaves (which also maintain the authoritative roster and capture UUIDs from the login lines), chat, advancements, and deaths (matched against the vanilla death message phrases, gated on the sender being in the roster so plugin noise and tellraw output can't spoof events).
- **In-game chat from the panel** - `ChatMessage` from the panel becomes a `tellraw @a` on stdin, so the chat bridge needs nothing game-side either.
- **Crash forensics** - exit code, the newest crash report under `crash-reports/`, and a capped excerpt, all reported before the container exits.
- **Signal forwarding and exit codes** - SIGTERM reaches java for the graceful world save, and the child's exit code is propagated so docker restart policies keep working.
- **Process telemetry** every 15 seconds: CPU and RSS from `/proc`, CFS throttling counters and the CPU quota from the cgroup (v1 and v2), and GC pauses parsed from the JVM's unified GC log, which is always enabled on Java 11+ (rotated, size-capped).

## The session to the panel

`proto/discopanel/agent/v1` defines one service with one RPC: a bidirectional `Session` stream. Telemetry flows up, console commands and chat messages flow down. The panel already serves Connect over h2c on its one port, so the stream needs no extra listener.

The container dials the panel, not the other way around. That survives container IP churn and panel restarts (the supervisor redials with capped backoff, and h2 pings detect half-dead connections). On reconnect it resends the ready state and the roster. When the panel is unreachable, telemetry is dropped rather than queued - the server always comes first.

Authentication is a per-server bearer token:

- Before each start, `lifecycle.writeAgentSpec` generates a fresh token, stores only its SHA-256 hash on the Server row, and writes the plaintext to `/data/.discopanel/agent.json` (mode 0600) along with the panel URL.
- The panel URL is `docker.agent_url` if set, otherwise auto-detected: the panel's own container IP on the shared bridge network when the panel runs containerized, or the network gateway when it runs on the host.
- The stream handler authenticates in-handler, because streaming RPCs bypass the unary auth interceptors and agent tokens are container credentials, not user identities. The first message must be a `Hello` whose server id matches the token.

A failure to write the spec is non-fatal. The server starts anyway and monitoring falls back to scraping.

## The disco-agent javaagent

`agent/` in the discoruntime repo is a single Gradle module producing `disco-agent.jar`: a `-javaagent` with a `Premain-Class`, Java 8 bytecode, and exactly one dependency (protobuf javalite, relocated under `app.discopanel.agent.libs` so it can never clash with a pack's own protobuf). It contains no Minecraft, mappings, loader, or version-specific code of any kind, which is the entire point: the supervisor adds one flag to the java command line and the same jar covers vanilla, Fabric, Quilt, NeoForge, Forge, Paper and friends, on any Java 8+ server ever released. Nothing is installed into `mods/` or `plugins/`, so there is no jar to version-match and no loader to support.

The agent stays dormant unless `-Ddiscopanel.agent.port` is set (i.e. the JVM was launched by the supervisor). When active it runs two samplers on daemon threads and ships results over loopback TCP port 25585, 4-byte length-prefixed protobuf frames with the same `AgentMessage` envelope as the panel session:

- **JvmSampler** - heap, GC pause deltas, thread and class counts from the standard MX beans.
- **TickSampler** - tick timing measured at the JVM scheduling level. Every Minecraft version runs its tick loop on one thread (named "Server thread" since forever) that alternates between tick work and waiting out each 50ms slot. The sampler polls that thread's state once per millisecond (about 0.05% of one core) and reports the busy share and the longest uninterrupted busy run per window.

The supervisor assembles the final tick numbers: below saturation the game's fixed 50ms cadence makes the busy share the mean tick cost (MSPT = 50 x busy fraction, TPS = 20, worst run = worst tick). A pegged thread hides tick boundaries, so there the supervisor uses the server's own overload confession - the "Can't keep up!" console line every version since alpha prints per 2000ms of tick debt, with the exact figure included on modern versions - and computes TPS from the debt rate. Folia has no single tick thread, so it reports no global TPS, which is the only honest answer for Folia.

The javalite messages are generated from this repo's agent protos by discoruntime's `make gen` via its `buf.gen.agent.yaml` into `agent/src/generated`. `make agent` there builds the jar in a containerized Gradle (run `make gen` first). Its `docker/Dockerfile.runtime` builds it in its own stage and bakes it into `/opt/discopanel/agent/disco-agent.jar`, generating the protos itself so image builds need no local `make gen`. Shipping an agent change means shipping runtime images.

## What flows

Up (agent to panel): `Ready`, `Stopping`, `Exited` (with crash report), `ProcSample` (CPU, RSS, throttling, GC log pauses), `TickSample` (TPS, MSPT avg/max), `JvmSample` (heap, GC pauses from MXBeans, threads), `PlayerEvent` (join, leave, death, advancement, chat), `Roster` (authoritative player list, resent every 30s and on reconnect).

Down (panel to agent): `ConsoleCommand` (written to java stdin) and `ChatMessage` (broadcast in game via tellraw).

## Precedence over scraping

Agent data lands in the same `ServerMetrics` store the SLP loop feeds, and while fresh it wins:

- Tick telemetry is the only TPS/MSPT source. RCON is never polled for metrics, it stays reserved for admin commands.
- A roster fresher than 90 seconds keeps SLP samples from clobbering player data (SLP truncates the sample) and turns off roster diffing - the hub emits exact `PLAYER_JOIN`/`PLAYER_LEAVE` events instead, plus the death/advancement/chat events scraping never had.
- The ready report counts as proof of life in the health record, so servers whose SLP is blocked go healthy immediately, and a live ready session keeps vouching while SLP pings fail.
- SLP keeps its jobs as the health probe and the source of MOTD, version, and favicon. Docker stats keep the whole-container CPU/memory view.

The degradation ladder:

1. Agent enabled: console events, tick and JVM telemetry, proc/cgroup stats, stdin console.
2. Agent disabled: SLP sampling and docker stats only.

Known limits, by design: chat events parse the vanilla `<name> message` console format, so a chat plugin that rewrites console output hides chat from the bridge; deaths with fully custom modded messages are missed. Both degrade to absent events, never to version- or loader-specific code.

## The autopilot

`internal/autopilot` turns the telemetry plus the server's configuration into the **health check**: plain-language findings, each with a one-click fix where one exists. It is pure analysis - reads the collector's cached state, and fixes just mutate a `ServerProperties` for the caller to persist. Served by `GetServerPerformanceReport` and `ApplyPerformanceFix` on the server service, rendered as the health strip of the Server Status card on the server dashboard.

The checks: heap vs container limit (warning when headroom is thin, critical when the heap can't fit), Aikar/MeowIce both enabled, ZGC suggestion for 12 GB+ heaps on Java 21+, CPU throttling (warning at 5% of CFS periods, critical at 20%), GC pauses (warning at 200 ms, critical at 1 s), tick health from MSPT (warning at 40 ms, critical at 50), heap over 92% full, and a crash within the last 24 hours.

Two enforcement points back the report up at start time:

- The **memory guardrail** (`lifecycle/guardrail.go`) clamps a heap that can't fit the container limit for that start only, logging to the server console. The stored config is untouched - the health check points at the real fix.
- `-XX:ActiveProcessorCount` is pinned to the CFS quota by the supervisor, so GC and compiler thread pools stop being sized for the host's cores.
