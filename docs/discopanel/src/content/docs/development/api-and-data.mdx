## Proto-first

Everything the frontend or an API consumer can do is declared in `proto/discopanel/v1/*.proto` - one file per domain (`server`, `config`, `task`, `module`, `proxy`, `auth`, `user`, `role`, `file`, `upload`, `mod`, `modpack`, `minecraft`, `event`, `websocket`, `support`, `common`). `make gen` regenerates the Go server stubs (`pkg/proto`) and the TypeScript clients (`web/discopanel/src/lib/proto`). Neither is ever edited by hand. If a behavior isn't reachable from a proto, it isn't part of the API.

An OpenAPI spec is generated from the same protos and served by `internal/rpc/handlers/openapi.go` - that's what the [API Reference](/api/) page renders.

## The RPC server

`internal/rpc.Server` is the single HTTP surface. It serves ConnectRPC over h2c (so gRPC, gRPC-Web and plain browsers share one port), plus gRPC reflection, the embedded SvelteKit build, the WebSocket hub, and the chunked upload/download handlers.

Service implementations live in `internal/rpc/services/`, one file per proto service. They follow a strict shape: validate the request, read/write through `internal/db.Store`, and delegate real work to the owning subsystem - `lifecycle.Manager` for anything that starts or stops a server, the scheduler for tasks, the module manager for sidecars. Handlers set an interim status and fire the lifecycle call in a goroutine (starts get a 2-hour timeout for provisioning, stops 15 minutes). The panel-side health checker settles the final status later. No docker calls in handlers.

## Auth and permissions

- `internal/auth` - JWT-backed sessions, local username/password login, and OIDC (issuer, client, role-claim mapping all in panel config). The OIDC callback is a plain HTTP handler, not an RPC.
- `internal/rbac` - a casbin enforcer on the same SQLite database (gorm adapter). Default policies are seeded at startup, and `auth.anonymous_access` seeds a more permissive set.

## WebSockets

`internal/ws.Hub` streams server console output and log lines to the browser and accepts console commands over the same socket. Commands go through `command.Sender` - the panel's native RCON client - and are accepted while a server is `running` **or** `unhealthy`, because a responsive RCON is proof of life even when pings fail. The provisioner's progress sink also feeds this stream, which is why installs show up in the server console prefixed `[provision]`.

## The database

`internal/db` is GORM on SQLite. `store.go` is the only query surface, models live in `models.go`, and `migrations.go` holds ordered, id-stamped migrations (e.g. `20260701_001_reset_itzg_image_tags`, which cleared stored itzg-era image tags so servers re-derive a runtime tag from their Minecraft version).

The interesting models:

- `Server` - identity, loader, MC version, memory, port, proxy hostname/listener, container ID, status. Status is a string enum (`stopped`, `provisioning`, `creating`, `starting`, `running`, `stopping`, `paused`, `unhealthy`, `error`) mapped 1:1 to the proto `ServerStatus`.
- `ServerConfig` - every per-server setting, one field per option. This struct is the workhorse. See below.
- `ScheduledTask` / `TaskExecution`, `Module` / `ModuleTemplate`, `ProxyListener`, `User` / `Role`, `Webhook` config blobs on tasks.

## The config field pipeline

`ServerConfig` fields carry struct tags that make one declaration serve four consumers:

```go
UseAikarFlags *bool   `json:"useAikarFlags" env:"USE_AIKAR_FLAGS" default:"false" desc:"Use Aikar's optimized JVM flags for GC tuning" input:"checkbox" label:"Use Aikar Flags"`
MOTD          *string `json:"motd" prop:"motd" default:"" desc:"Server log in message" input:"text" label:"Message of the Day"`
```

| Tag | Consumer |
|---|---|
| `label`, `desc`, `input`, `default` | `buildConfigCategories` in the config service renders the UI from these - the frontend has no hardcoded form. |
| `env` | JVM/runtime settings only. Reflected into container env vars at create time and read by the runtime entrypoint. |
| `prop` | server.properties keys. The provisioner writes these on every start, merged so unmanaged keys survive. |
| `ephemeral` | Cleared automatically after a successful start (`ForceProvision` is the example). |

The split is deliberate: `env:` fields configure the JVM and cross into the container as environment, while `prop:` fields configure the game and cross as files. A field has one or the other, never both.

### Adding a per-server setting

1. Add the field to `ServerConfig` in `internal/db/models.go` with the right tags. Pointer types, so nil means "unset".
2. If it's a select, register its options in the config service, and add it to the right category index there too.
3. Consume it where it matters - the provisioner for files, `buildEnvFromConfig` picks up `env:` tags automatically, lifecycle/idle for behavior settings.
4. `make gen` only if you also touched protos (plain config fields usually don't).

The UI, defaults handling, and persistence come for free.

## Status flow

`proto/discopanel/v1/common.proto` defines `ServerStatus`, and `dbStatusToProto` in the server service maps the DB strings. Three layers write it, in order:

1. The RPC handler sets an interim status (`provisioning` for starts, `stopping` for stops) and returns immediately.
2. The lifecycle manager walks the machine: `provisioning`, then `creating`, then `starting`.
3. The metrics collector's pings settle the final state, `running` or `unhealthy`.

The frontend badge styling keys off the proto enum, including `PROVISIONING` (yellow, pulsing) and `PAUSED` (blue "Sleeping").
