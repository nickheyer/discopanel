## The bus

`internal/events.Bus` is the single egress point for server events - one synchronous fan-out, no queues, no goroutines. `Emit` calls every subscriber in registration order, and handlers must return promptly (anything slow spawns its own goroutine). There is exactly one bus, wired in `cmd/discopanel/main.go`. Don't add side channels. If a subsystem needs to react to a server event, it subscribes here.

Event types are the proto enum `TriggeredEventType`, so the same values flow through the API, the DB (event-triggered task subscriptions), and webhook payloads:

| Event | Emitted by | When |
|---|---|---|
| `SERVER_START` | lifecycle manager | container started successfully |
| `SERVER_STOP` | lifecycle manager | graceful stop completed |
| `SERVER_RESTART` | lifecycle manager | after a restart (in addition to STOP + START) |
| `SERVER_HEALTHY` | metrics collector | first successful ping this run, or recovery from unhealthy |
| `PLAYER_JOIN` / `PLAYER_LEAVE` | agent hub or metrics collector | exact events parsed from the server console by the runtime supervisor when the agent session is live, roster diff between polls otherwise. Carries `Data["player"]` |
| `PLAYER_DEATH` / `PLAYER_ADVANCEMENT` / `PLAYER_CHAT` | agent hub | parsed from the server console by the runtime supervisor, so they need the [runtime agent](/development/agent/). `Data["detail"]` carries the death message, advancement title, or chat line |

Without the agent, the collector derives join/leave by diffing the player-name roster (from SLP samples or RCON `list`) every 10 seconds, seeding baselines at startup so already-running servers don't emit a storm on panel boot. While an agent session is live the diffing is suppressed - the supervisor reports the exact events, so diffing too would double-emit.

## Subscribers

1. **Lifecycle manager** - runs the user-configured RCON command lists: `RCON Commands on Startup` on SERVER_HEALTHY, on-connect/first-connect on PLAYER_JOIN, on-disconnect/last-disconnect on PLAYER_LEAVE. It tracks a per-server roster to know "first" and "last".
2. **Module manager** - starts auto-start modules on SERVER_START, stops lifecycle-following modules on SERVER_STOP, and dispatches per-module event hooks for everything.
3. **Scheduler** - looks up event-triggered tasks (`ListEventTriggeredTasks`) and executes each one with the firing event attached.

## The scheduler

`internal/scheduler` owns `ScheduledTask` / `TaskExecution`. Task types: `command` (RCON), `backup` (see `backup.go` - save-off, flush, zip, retention pruning), `restart`/`start`/`stop` (one-line calls into the lifecycle manager), `script` (docker exec inside the server container), and `webhook`.

Schedules: cron expression, fixed interval, one-shot at a time, or **event** - no next-run time, executions arrive only through the bus subscription. Executions record trigger provenance (`scheduled`, `manual`, `event`), output, retries, and duration.

`require_online` skips a task when the server isn't running - except webhook tasks, which always fire: they notify rather than operate on the server, and the most useful events (`server_stop`) happen precisely when the server isn't running.

## Webhooks

`internal/webhook` is delivery only. The scheduler decides when. A webhook task's JSON config carries URL, optional HMAC secret, custom headers, payload template, and retry/timeout settings.

- Payload: `{event, timestamp, server: {id, name, status, mc_version, mod_loader, players_online, max_players, port}, data}` by default, or the task's Go `text/template` rendered against the same struct. Event names are lowercase (`server_start`, `player_join`, ...). Manual runs send `manual`.
- Headers: `Content-Type: application/json`, `User-Agent: DiscoPanel-Webhook/1.0`, `X-DiscoPanel-Event`, `X-DiscoPanel-Delivery` (UUID per attempt batch), and `X-DiscoPanel-Signature: sha256=<hex hmac>` when a secret is set. Custom headers apply last and can override.
- Retries: exponential backoff from `retry_delay_ms` (default 1s), `max_retries` extra attempts, per-attempt timeout default 5s. 2xx counts as success, and the response body (first 4 KB) lands in the execution record either way.

The player count in payloads is refreshed from the metrics collector at delivery time, so join/leave notifications carry accurate numbers.

## Modules

Modules are sidecar containers attached to a server: Geyser (Bedrock crossplay), BlueMap (web map), the Prometheus exporter, the status panel. `internal/module.Manager` creates them from `ModuleTemplate` rows. The built-ins are upserted at startup in `builtin_templates.go` (which also deletes the obsolete itzg-era backup/rcon templates when nothing references them).

Templates declare image, ports, default env/volumes/memory, health check, and optional init commands. Values support `{{...}}` placeholders (`internal/alias`) expanded per instance: `{{server.id}}`, `{{server.data_path}}`, `{{module.ports.<Name>.host_port}}`, `{{host.uid}}`, `{{config.server.port}}` and friends - that's how the Geyser template points itself at `discopanel-server-{{server.id}}:25565` without user input. Host ports are allocated from `module.port_range_min/max` (default 8100-8199). HTTP module ports can be exposed through the proxy.

Module containers follow the same rules as server containers: `discopanel.managed=true` labels, the `discopanel-network` bridge, logs streamed through the same log streamer.
