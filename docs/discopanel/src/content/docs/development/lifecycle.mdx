## The lifecycle manager

`internal/lifecycle.Manager` is the only component allowed to move servers between states. RPC handlers, scheduler tasks, boot auto-start, config apply and the idle watcher all call it. Methods: `Start`, `Stop`, `Restart`, `Recreate`, `Pause`, `Wake`. Per-server start locks collapse concurrent start requests into one.

A start walks the states in order. Everything else is a single hop:

| From | To | When |
|---|---|---|
| stopped | provisioning | start requested, files installed or verified |
| provisioning | creating | container created, or recreated on image mismatch |
| creating | starting | container running, waiting on the first ping |
| starting | running | the panel's health checks pass |
| provisioning / creating / starting | error | any step failed |
| running | stopping | stop requested |
| stopping | stopped | container exited |
| running | paused | autopause froze the container |
| paused | running | a player join woke it |

Stop details worth knowing: paused containers are resumed first, since a frozen process can't handle SIGTERM. An optional in-game announcement (Stop Server Announce Delay) runs before the stop. The SIGTERM grace window is the server's Stop Duration (default 60s), after which the container is killed. `Restart` is `Stop` plus `Start`, which is also how config changes get applied, since the provisioner rewrites config files during every start.

On start, if the existing container's image doesn't match the required runtime image (the Java requirement changed after an MC upgrade), the container is recreated automatically.

## Health

The runtime image has no Docker HEALTHCHECK. Health is computed by the panel:

- The metrics collector pings every running container with a real Server List Ping (every 15s by default) and records the results per container run, keyed to the container's StartedAt so restarts reset cleanly.
- A responsive RCON also counts as proof of life. This matters for mods that firewall unknown status pings (the classic Connectivity mod false-unhealthy): SLP can be blocked while the server is perfectly playable.
- `GetContainerStatus` consults the record through the `docker.HealthChecker` interface:

| Record state | Reported status |
|---|---|
| No successful ping yet, within the 15 minute startup grace | starting |
| No successful ping yet, grace expired | unhealthy |
| Was healthy this run, fewer than 3 consecutive failures | running |
| Was healthy this run, 3 or more consecutive failures | unhealthy |
| No data for this run but the container has been up longer than the grace (panel restart) | running, until pings say otherwise |

`SERVER_HEALTHY` fires on the transition to healthy and drives the RCON Commands on Startup setting. Unhealthy means "not answering pings", not "down": unhealthy servers still accept console commands and still get polled.

## The metrics collector

One component, three roles: metrics source for the UI, `docker.HealthChecker` (above), and `lifecycle.PlayerCounter` for the idle watcher. Five loops, all panel-side:

| Loop | Default interval | Gathers |
|---|---|---|
| Docker stats | 5s | CPU% and memory via the engine's stats API |
| SLP | 15s | players, MOTD, version, latency, favicon. The primary health evidence |
| RCON | 10s | player roster via `list` when SLP data is stale, plus TPS via the loader's TPS command. Also counts as health |
| Disk | 60s | data dir and world sizes, disk free |
| Event derivation | 10s | SERVER_HEALTHY transitions and PLAYER_JOIN/LEAVE roster diffs onto the bus |

`PlayersOnline(serverID)` answers `(count, known)` and only claims `known` from SLP data fresher than two intervals - the idle watcher never acts on stale numbers.

## Autopause internals

The idle watcher (a 30s loop in `lifecycle/idle.go`) tracks last activity per server from fresh SLP player counts. Unknown player data means no action is ever taken. The timeouts distinguish "nobody ever joined since start" from "players left":

| Policy | Established (players left) | Initial (nobody ever joined) |
|---|---|---|
| Autopause | 3600s | 600s |
| Autostop | 3600s | 1800s |

Autopause is silently skipped for servers without a proxy hostname - pause is useless without the proxy's wake gate. Both policies can be enabled together: the server pauses first, and a paused server can still hit the autostop timeout and shut down fully.

Pause is `docker pause` (the cgroup freezer). The proxy is the wake path, through the `proxy.ServerGate` interface that the lifecycle manager implements. The proxy already parses the Minecraft handshake for hostname routing, so it knows what every connection wants:

- A status ping (`NextState=1`) for a paused server gets a synthesized response from the proxy itself ("sleeping, join to wake", echoing the client's protocol version). The container stays frozen.
- A login (`NextState=2`) triggers `WakeServer`, which unpauses the container, and the proxy dials the backend with retries for about 10 seconds while the JVM resumes.

The paused check on the connection hot path is an in-memory set, and proxy routes are kept alive for paused servers.

## The end-to-end test

`internal/lifecycle/e2e_test.go` boots a real server through the whole stack: provision, container, SLP health, RCON, graceful stop. It's skipped unless explicitly enabled.

```sh
make runtime-local-21    # once, to have a local runtime image
DISCO_E2E=1 go test ./internal/lifecycle -run TestE2EVanillaServer -v -timeout 20m
DISCO_E2E=1 DISCO_E2E_LOADER=fabric go test ./internal/lifecycle -run TestE2EVanillaServer -v
```

`DISCO_E2E_MC_VERSION` overrides the Minecraft version. A vanilla run takes about 12 seconds on a warm network. Run it before releases and whenever you touch the provisioner, the docker client, the entrypoint or the health code.
