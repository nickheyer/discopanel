These pages are for people working on DiscoPanel itself. If you just run servers with it, you don't need anything in here.

## The system at a glance

DiscoPanel is one Go process. It embeds the web UI, keeps its state in SQLite, talks straight to the Docker Engine API (no compose, no swarm), and runs every Minecraft server in a container built from a minimal `discoruntime` image.

The design rule that shapes everything: **containers hold nothing but Java and a small supervisor**. Downloading server software, installing packs, writing config files, health checking, autopause - all of it happens in the panel process, where it can be unit tested and reused. The itzg image DiscoPanel historically used did the opposite: install scripts, config generation, healthchecks and autopause all lived inside the container. The migration moved that logic panel-side and shrank the container to a JRE. The runtime image is not supported for standalone use. Without a provisioned data directory it refuses to start.

## How a request flows

The SvelteKit UI calls the API through generated ConnectRPC clients. `internal/rpc.Server` serves everything on the panel's one HTTP port (h2c, so gRPC and browsers share it): the RPC services, the embedded frontend, the WebSocket hub, and the upload/download/OpenAPI handlers. Services validate and persist, then delegate the real work - starting a server is `lifecycle.Start` in a goroutine, never docker calls inside a handler. Progress and console output stream back to the browser through the WebSocket hub.

Startup wiring in `cmd/discopanel/main.go` builds the graph in dependency order:

1. store, docker client, event bus
2. metrics collector, registered as the docker client's health checker
3. agent hub, feeding the collector and the bus, and wired into the command sender as the RCON fallback
4. provisioner
5. lifecycle manager, registered as the proxy's server gate and given the collector as its player counter
6. proxy, scheduler, module manager
7. RPC server

The bus subscribers are the module manager, the scheduler, and the lifecycle manager itself.

## What a start looks like

1. `lifecycle.Start` marks the server as provisioning and calls `provisioner.Ensure`.
2. Ensure compares `.discopanel/manifest.json` against the desired loader, Minecraft version and pack pin. If they match, nothing is downloaded. Otherwise it installs the difference and rewrites `launch.json` and the manifest.
3. Config files (`server.properties`, `eula.txt`, ops, whitelist, icon) are regenerated from the DB on every start, matched or not. So is `agent.json`, the runtime agent's connection spec with a fresh token. A heap that can't fit the container memory limit is clamped for this start.
4. The container is created if missing, or recreated if its image no longer matches the required Java version.
5. Docker starts it. The entrypoint reads `launch.json`, attaches the disco-agent javaagent, and runs java as a supervised child. `SERVER_START` goes out on the event bus.
6. The agent's ready report (or the collector's pings, whichever lands first) flips the status to running. See [Lifecycle & Health](/development/lifecycle/).

Every path that starts or stops a server (RPC handlers, scheduler tasks, boot auto-start, idle policies, config apply) goes through the lifecycle manager. Don't drive `internal/docker` directly for lifecycle work.

## Orchestration

Plain Docker Engine API. No compose, no swarm.

- One container per server, named `discopanel-server-<id>` and labeled `discopanel.managed=true`. Orphans are cleaned up at panel startup.
- One bind mount, the server folder to `/data`.
- Everything joins the `discopanel-network` bridge. Non-proxied servers bind their game port on the host and RCON on `127.0.0.1:<port+10>`. Proxied servers expose no host ports at all. The proxy dials the container IP instead. Details in [Proxy & Networking](/development/proxy/).
- Env vars carry JVM settings only, built by reflection over `env:` tags on `ServerProperties`. Game settings cross the boundary as files.
- Forge, NeoForge and Quilt installers need Java, so the provisioner runs them in a throwaway container of the same runtime image (`docker.RunOneShot`) with `/data` mounted, streams the output to the server console, and removes it afterwards.
- Restart policy is `unless-stopped`, so servers come back after a host reboot even if the panel doesn't.

## Events

`internal/events.Bus` is a small synchronous fan-out. The lifecycle manager emits `SERVER_START`, `SERVER_STOP` and `SERVER_RESTART`, the collector derives `SERVER_HEALTHY` from ping data, and player events come from the agent hub when a session is live (exact join/leave plus death, advancement and chat) or from the collector's roster diffing when not. Subscribers are the module manager (auto-start/stop of sidecars), the scheduler (event-triggered tasks and webhooks), and the lifecycle manager itself, which runs the user-configured RCON startup/connect/disconnect commands. Full detail in [Events & Automation](/development/events/).
