The provisioner (`internal/provisioner`) turns a server row plus its config into a ready-to-launch data directory. It runs in the panel process, takes a per-server lock so a server is never provisioned twice at once, and streams progress lines to the server console.

## Ensure

`Ensure(ctx, server, config)` runs on every start and is idempotent:

1. Read `.discopanel/manifest.json` and compute the desired state (loader, MC version, modpack source and pin).
2. Install only if something changed: no manifest, loader or MC version differs, pack identity or pin differs, the launch target is missing from disk, or the ephemeral `ForceProvision` flag is set.
3. On install, write `launch.json` and a fresh manifest.
4. Always rewrite the config files: `server.properties` from `prop:`-tagged `ServerProperties` fields (merged, so unmanaged keys survive), `eula.txt`, the icon, `ops.json` and `whitelist.json` (Mojang UUID lookups online, derived UUIDs for offline mode), plus any declared Modrinth projects.
5. Return the resolved loader, loader version, MC version and Java major. The lifecycle manager writes these back to the DB (packs are authoritative for the MC version) and picks the container image from the Java major.

RCON is always forced on in `server.properties` with a per-server default password. The console, metrics and backups depend on it.

## The launch spec

`pkg/runtimespec` is the only interface between panel and container: plain JSON in `<data>/.discopanel/`, which keeps the entrypoint stdlib-only.

```json
{
  "version": 1,
  "kind": "jar",
  "jar": "fabric-server-launch.jar",
  "no_gui": false,
  "loader": "fabric",
  "mc_version": "1.21.1",
  "java_major": 21
}
```

Three kinds. `jar` runs `java <flags> -jar <jar> [extra args] nogui`. `args-file` runs `java <flags> @<file> nogui` (modern Forge and NeoForge, with `args_file` pointing into `libraries/`). `custom` passes the user's `exec` string through verbatim.

Everything the provisioner tracks lives in `<data>/.discopanel/`:

| File | Purpose |
|---|---|
| `launch.json` | the launch spec above, read by the container entrypoint |
| `manifest.json` | what's installed (loader, versions, pack pin) - the idempotency record |
| `installers/` | downloaded installer jars and pack archives, per-server scratch |
| `blocked-mods.json` | the current list of mods needing manual download, if any |

## Where server software comes from

| Loader | Upstream | Notes |
|---|---|---|
| Vanilla | piston-meta.mojang.com | SHA-1 verified. Also the source of the Java requirement. |
| Fabric | meta.fabricmc.net/v2 | Prebuilt launcher jar. The vanilla jar is pre-seeded so first boot is offline. |
| Quilt | meta.quiltmc.org/v3 | No prebuilt launcher exists, so the installer runs in a one-shot container. |
| Forge | promotions_slim.json + maven | Recommended, then latest. Exact maven dir resolved from maven-metadata.xml because old versions carry branch suffixes. Installer runs one-shot. Result detected as unix_args.txt (modern) or a runnable jar (legacy). |
| NeoForge | maven.neoforged.net | Version prefix derived from the MC version (1.X.Y becomes X.Y., date-based 26.A.B becomes 26.A.B.). MC 1.20.1 lives under the legacy artifact. SHA-256 sidecars verified. |
| Paper / Folia | fill.papermc.io/v3 | The old api.papermc.io v2 is dead (HTTP 410). SHA-256 verified. |
| Purpur | api.purpurmc.org/v2 | Latest successful build, MD5 verified. |

Anything else falls through to a clear error pointing at the custom jar path. Downloads are atomic (temp file, then rename) and checksum-verified whenever the upstream publishes digests. Large downloads report throttled byte progress to the server console, and modpack file sets download eight at a time. All requests go out with the panel's configured User-Agent.

### The artifact cache

Every download carrying a strong checksum (SHA-1/256/512; MD5 is verified but never trusted as identity) routes through a content-addressed cache at `{data_dir}/cache/cas/`, keyed by the digest the upstream already published. The second server installing the same pack, and every reinstall, materializes its mods, jars, and installers from disk instead of the network — which also keeps CurseForge rate limits at bay. Entries are reflinked when the filesystem supports it (copied otherwise, so an edit inside a server dir can never corrupt the shared copy), hits refresh an entry's timestamp, and anything untouched for 45 days is pruned in the background at the next install.

Forge and NeoForge get one more layer: their installers download a maven closure and run processors into `libraries/`, none of which passes through our downloader. After a successful install the tree is archived under `cache/libtrees/` keyed by loader version; the next install of that version pre-seeds `libraries/` from the archive and still runs the installer, which validates existing files by hash and finishes in seconds instead of minutes. The installer stays the source of truth; the archive only saves it the work.

## Modpacks

**Modrinth**: resolve the project and version (pinned, or newest allowed by the release-channel setting), download the primary `.mrpack` (SHA-512), parse `modrinth.index.json`, download its files while skipping anything marked server-unsupported or user-excluded, apply `overrides/` then `server-overrides/` without clobbering existing files, and install the loader the pack declares in `dependencies`.

**CurseForge**: resolve the slug or URL, prefer the author's server pack when one exists (it's curated for headless use and sidesteps client-only mods entirely), otherwise use the client pack's manifest: apply overrides, bulk-fetch file and mod metadata, skip files CurseForge marks client-only, download each mod into the right directory for its class (mods, resourcepacks, shaderpacks, plugins, datapacks), then install the declared loader. Server packs without a manifest get extracted wholesale and the launch target is detected, including running a bundled installer jar if that's what the pack shipped.

**Blocked mods**: when a mod's author has opted out of API distribution, the provisioner collects every affected mod, writes `.discopanel/blocked-mods.json`, and fails with an error listing file names and download pages. Files already sitting in `mods/` are skipped, so the manual download, upload, retry loop converges.

## Adding a new loader

1. Write an installer in `internal/provisioner/`. Copy the shape of `installPurpur` for a direct download or `installQuilt` for an installer-driven one. Resolve the version, download with a checksum, finish with `finishLaunch` and the right launch kind.
2. Route it in the `install()` switch in `provisioner.go`.
3. If modpacks can declare it, map it in `installPackLoader`.
4. No image work. Loaders are data-directory content, images only carry Java.
5. Prove it with the e2e test: `DISCO_E2E=1 DISCO_E2E_LOADER=<loader> go test ./internal/lifecycle -run TestE2EVanillaServer -v`
