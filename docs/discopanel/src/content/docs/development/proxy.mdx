## The proxy

`internal/proxy` is a Minecraft-protocol-aware TCP proxy inside the panel process. It parses the handshake packet of every incoming connection, reads the hostname the client typed, and pipes the connection to the matching server's container. That's what lets every server share one port.

**Listeners.** Each `ProxyListener` row is one listening port, running its own `MinecraftProxy` instance. A default listener is created at startup on the configured `proxy.listen_port`. More can be added, with ports allocated from `proxy.port_range_min/max` when none is given explicitly. A server binds to exactly one listener via `ProxyListenerID`, and hostnames are unique per listener (DB unique index), so the same hostname can exist on two ports.

**Routes.** A route maps a hostname to a server. The default hostname is derived from the server name - lowercased, spaces to dashes, suffixed with `proxy.base_url` - and users can override it per server in the Routing tab (bare names get `base_url` appended). Routes are registered when a server starts, removed when it stops, and deliberately **kept while a server is paused** so sleeping servers stay reachable.

## Sleep and wake

The proxy is the wake path for [autopause](/development/lifecycle/#autopause-internals), through the `proxy.ServerGate` interface that the lifecycle manager implements. On every routed connection the proxy asks the gate whether the target is sleeping:

- **Status ping** (`NextState=1`): the proxy answers the ping itself - a synthesized status response echoing the client's protocol version, version name `Sleeping`, zero players online, the server's real MOTD ("... sleeping - join to wake") and max players, plus the usual ping/pong exchange. The container stays frozen, and refreshing the server list never wakes anything.
- **Login** (`NextState=2`): the proxy calls `WakeServer` (15s budget) to unpause the container, then dials the backend with retries - 250ms apart for up to 10 seconds - because a just-thawed JVM needs a moment to accept connections.

The paused check on the connection hot path is an in-memory set on the lifecycle manager, not a DB read.

## Container networking

All managed containers join the `discopanel-network` bridge (`EnsureNetwork` creates it, labeled `discopanel.managed=true`). When the panel itself runs in a container (detected via `/.dockerenv`), it attaches itself to that network so it can reach the servers.

| | Game port | RCON |
|---|---|---|
| **Proxied server** | no host binding, the proxy dials `<container-ip>:25565` on the bridge | no host binding, the panel dials the container IP |
| **Non-proxied server** | bound on `0.0.0.0:<server.Port>` | bound on `127.0.0.1:<port+10>` |

Inside the container the game port is always 25565 for proxied servers - the provisioner writes `server-port` into `server.properties`, so no env override exists. Additional per-server ports (voice chat, map mods) bind `0.0.0.0` with their configured protocol and are not proxied.

The metrics collector's SLP pings and the panel's RCON client use the same bridge-IP path as the proxy, which is why none of it requires host ports.

## Host path translation

The panel bind-mounts each server's data directory into its container at `/data`. When the panel itself is containerized, the path it sees is not the path the Docker engine needs, so every mount goes through `TranslateToHostPath`:

- `DISCOPANEL_DATA_DIR` - where the panel sees its data inside its own container (default `/app/data`).
- `DISCOPANEL_HOST_DATA_PATH` - the host path that directory is mounted from.

Both must describe the same volume entry. Running directly on a host, neither is set and paths pass through untouched. One-shot installer containers and module volume mounts go through the same translation.
