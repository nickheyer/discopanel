`discoruntime` (published as `ghcr.io/nickheyer/discoruntime`) is the image every Minecraft server runs on.

## Contents

The runtime sources (supervisor, javaagent, Dockerfile) live in the discoruntime repo, which builds against this repo's protos and `pkg/`. `docker/Dockerfile.runtime` there has four stages:

1. `bufbuild/buf` regenerates the protobuf code (Go for the entrypoint, javalite for the javaagent) with the same recipe as `make proto`, so image builds work from a clean checkout - no `make gen` first, no generated code in the build context.
2. `golang:alpine` compiles `cmd/runtime` (statically linked, `pkg/runtimespec` plus the generated agent protos).
3. `gradle:9-jdk21` builds the disco-agent javaagent jar from `agent/`.
4. `eclipse-temurin:<JAVA_VERSION>-jre` plus `ca-certificates` and `tzdata`, with the compiled binary as the entrypoint and the javaagent at `/opt/discopanel/agent/disco-agent.jar`. Temurin's Ubuntu base was picked over Alpine because mod natives expect glibc.

The first three stages are pinned to `--platform=$BUILDPLATFORM`: generated code and jars are architecture independent, and the Go entrypoint cross-compiles via `TARGETOS`/`TARGETARCH`. A multi-arch build therefore runs the heavy toolchains once, natively, and only the final JRE stage is per-architecture. This is load-bearing, not just a speedup: the forge and neoforge builds break under QEMU emulation (NeoForm can't resolve its own JVM through `ProcessHandle` there). A default of `linux/amd64` on the `BUILDPLATFORM` arg keeps the Dockerfile buildable on the legacy non-BuildKit builder, where the variable is otherwise undefined.

That's the whole image. The entrypoint reads `/data/.discopanel/launch.json`, fixes `/data` ownership, builds the java argv from env vars (heap, flag sets, JMX, user options), and runs java as a supervised child under the configured UID/GID. The entrypoint stays PID 1: it forwards SIGTERM so the world saves during the graceful window (the server's Stop Duration setting, 60s by default), propagates java's exit code so docker restart policies keep working, watches the console for readiness and crashes, feeds console commands to java stdin, and streams telemetry to the panel. The supervisor half of the story is on [Runtime Agent](/development/agent/).

## The env contract

Environment variables are the container's entire config surface besides `launch.json`, and they carry JVM concerns only. The panel builds them by reflection over `env:` tags on `ServerProperties`. The entrypoint consumes:

| Variable | Effect |
|---|---|
| `UID`, `GID` | ownership fix + privilege drop (default 1000/1000) |
| `MEMORY`, `INIT_MEMORY`, `MAX_MEMORY` | `-Xms`/`-Xmx`. `MEMORY` sets both, the others override individually |
| `AUTO_MEMORY` | replaces fixed `-Xms`/`-Xmx` with `-XX:InitialRAMPercentage=50` and `-XX:MaxRAMPercentage=75`, so the JVM sizes its heap from the cgroup memory limit |
| `USE_AIKAR_FLAGS` | Aikar G1GC set, switching to the large-heap variant at 12 GB max heap or more. Applied by default when neither flag-set variable is present |
| `USE_MEOWICE_FLAGS` | MeowIce set behind `-XX:+IgnoreUnrecognizedVMOptions`, plus transparent huge pages, NUMA, and the vector module on Java 16+ |
| `USE_ZGC_FLAGS` | generational ZGC instead of G1. Java 21+ only, falls back to the G1 sets below it with a console warning. Wins over both G1 sets |
| `USE_FLARE_FLAGS`, `USE_SIMD_FLAGS` | Flare profiler flags, and `jdk.incubator.vector` (Java 16+) |
| `ENABLE_JMX`, `JMX_HOST` | remote JMX on port 7091 with no auth/ssl. `JMX_HOST` sets the RMI hostname |
| `TZ` | `-Duser.timezone` |
| `JVM_OPTS`, `JVM_XX_OPTS`, `JVM_DD_OPTS`, `EXTRA_ARGS` | raw passthrough. `JVM_DD_OPTS` entries get `-D` prefixed and `EXTRA_ARGS` goes after the jar |

A few args are added without any env var: `-Dlog4j2.formatMsgNoLookups=true` (Log4Shell), unified GC logging to `/data/.discopanel/gc.log` on Java 11+ (rotated and size-capped, the supervisor tails it for pause telemetry), and `-XX:ActiveProcessorCount` pinned to the CFS quota when the container has one and the user hasn't set it themselves, so GC and compiler threads aren't sized for the host's cores. Three performance defaults ride along the same way, each skipped when the raw opts already mention the matching flag family: `-XX:+UseTransparentHugePages` (the JVM silently falls back when the host has THP off; the report card flags hosts set to `never`), `-XX:+UseCompactObjectHeaders` on Java 25+ (smaller object headers, less heap and fewer GCs), and a self-maintaining class data archive on Java 19+ (`-XX:+AutoCreateSharedArchive` writing `/data/.discopanel/cds.jsa`, which shaves class loading on every restart and rebuilds itself when the JDK or classpath changes). The agent connection itself is not configured by env - the supervisor reads `/data/.discopanel/agent.json`, written by the panel before each start (see [Runtime Agent](/development/agent/)).

Beyond JVM flags, the supervisor also raises the scheduling priority of the server tick thread (`nice -10` on the `Server thread` task) once it appears, so GC, chunk IO, and other containers yield to the tick loop under CPU contention. Server containers are created with the `SYS_NICE` capability for exactly this; when a docker override drops it, the supervisor warns once and moves on. It also samples cgroup pressure stall information (PSI) every telemetry window, giving the panel early warning of memory exhaustion and disk stalls that plain usage numbers cannot see.

Booleans accept `true`/`1`/`yes`, case-insensitive. The image exposes 25565 (game) and 25575 (RCON) and declares `/data` as its volume. There is no HEALTHCHECK - health is panel-side, see [Lifecycle & Health](/development/lifecycle/#health).

## Tags

Tags map to Java major versions and nothing else: `java8`, `java11`, `java17`, `java21`, `java25`. Minecraft versions, loaders and packs are data-directory concerns and never appear in tags.

One opt-in variant exists per modern major: `java21-graal` and `java25-graal` swap the Temurin JRE for Oracle GraalVM (redistributable under the GFTC license), whose Graal JIT often ticks measurably faster on modded servers. The image sets `DISCO_JVM_VARIANT=graal` so the entrypoint can adapt: it adds `-XX:+EagerJVMCI` plus inliner tuning, and skips compact object headers (unverified with the Graal compiler). Pick the variant per server in settings; treat it as something to benchmark per pack, not a universal win.

Selection is automatic. `docker.RequiredJavaMajor` reads the required Java major from Mojang's version metadata and rounds up to the nearest published tag. MC 1.17 asks for Java 16 and runs on `java17`, which is the industry norm since Temurin never shipped a 16 JRE. Per-server overrides are validated against the published set, and a DB migration cleared the old itzg-era tags.

## The single sync point

The supported Java list lives in `pkg/javaversions` and every consumer reads it: `internal/docker/images.go` aliases the slices, and discoruntime's Makefile and release workflow shell out to `cmd/runtime -versions`.

## Shipping a new Java version

New variants are only needed when Mojang bumps the Java requirement.

1. Check that `eclipse-temurin:<N>-jre` exists on Docker Hub.
2. Add `N` to `pkg/javaversions`.
3. `make runtime` in the discoruntime repo, then run the e2e test against a Minecraft version that needs it.
4. Publish. Existing servers don't change. New and upgraded ones resolve the new tag automatically.

Keep old tags published even after dropping them from the list. Servers may still pin them.

## Building and publishing

- `make runtime` in the discoruntime repo builds every variant locally when its inputs changed, no push.
- Pushing a `v*` git tag on discoruntime (or dispatching its release workflow) builds multi-arch (amd64 and arm64) in CI and pushes every variant.

The image contents change so rarely that the Java major is the whole version story. If the entrypoint's behavior ever changes, that ships as new runtime images from discoruntime; the panel-side contract (`pkg/runtimespec` and the agent protos) stays in this repo.

Private mirrors: set `docker.runtime_image` in the panel config to another repository carrying the same tags. The panel pulls an image only when it's absent locally, so starts never block on the registry. Already-present images are re-pulled in the background (at most hourly) to pick up tag updates, and local-only images (unpublished dev builds) work because presence is checked before any pull.
